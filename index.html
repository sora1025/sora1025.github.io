requestAnimationFrame

Posted on 2020-03-01

一、前言

最近了解到这样一个做动画的API，过往只接触过JS中用定时器实现动画，或是CSS3中实现animation或者transition来实现。那么在C3动画如此强大的今天，这个API到底有什么用呢？

requestAnimationFrame，顾名思义就是请求动画帧。在学习这个API之前，我们要先了解几个概念：

  • 屏幕刷新频率：屏幕每秒出现图像的次数。普通笔记本为60Hz
  • 动画原理：计算机每16.7ms刷新一次，由于人眼的视觉停留效应，所以看起来是流畅的移动。

我们使用setTimeout制作动画时，是通过设定间隔时间来不断改变图像位置，从而达到动画的效果。但是很容易出现卡顿、抖动的现象。原因是：

  1. setTimeout被放入异步队列，当主线程空闲并且当设置的时间到达时，才会执行。因此实际执行时间总是比设定时间要晚。这一部分涉及到JS事件循环的知识，这里不再扩展。
  2. 刷新频率受屏幕分辨率和屏幕尺寸的影响。因此不同设备的刷新频率不同。setTimeout设置的时间间隔不一定和屏幕刷新时间相同，会引起丢帧。

归纳一点最关键的问题：开发者并不知道下一刻绘制动画的最佳时机是什么时候。

三、使用优势

  1. CPU节能

使用setTimeout实现的动画，当页面被隐藏或者最小化时，setTimeout仍然会在后台执行动画任务，由于此时页面处于不可见或不可用状态，刷新动画是没有意义的，完全是浪费CPU资源。

而requestAnimationFrame，当页面处于未激活的状态下，该页面的屏幕刷新任务也会被系统暂停，因此跟着系统系统走的requestAnimationFrame也会停止渲染，当页面被激活时，动画就从上次停留的地方继续执行，有效节省了CPU、GPU和内存的使用。

不过有一点需要注意，requestAnimationFrame是在主线程上完成的。这意味着，如果主线程非常繁忙，requestAnimationFrame的动画效果会大打折扣。

  1. 函数节流

在高频率事件(resize,scroll等)中，为了防止在一个刷新间隔内发生多次函数执行，使用requestAnimationFrame可保证每个刷新间隔内，函数只被执行一次，这样既能保证流畅性，也能更好的节省函数执行的开销。

三、How to use?

  1. 语法

    window.requestAnimationFrame(callback);

  • 参数

callback

requestAnimationFrame接受一个回调函数作为参数。这个回调函数会在浏览器重绘之前调用。

  • 返回值

requestID = requestAnimationFrame(callback)

返回一个整数，表示定时器的编号。这个值作为参数可传递给cancelAnimationFrame用于取消这个函数的执行。

二、Babel是啥？

Babel是一个JavaScript编译器，它把最新版的JavaScript编译成当下可执行的版本，简言之，利用Babel就可以让我们在当前的项目中随意的使用这些最新的ES6，甚至ES7的语法！

简单来说，Babel可以转译ES 2015+的代码，使它在旧的浏览器或者环境中也能够运行。

Babel的功能很纯粹。我们传递一段源代码给Babel，然后它返回一串新的代码给我们。就是这么简单，它不会运行我们的代码，也不会去打包我们的代码。它只是一个编译器。

三、Babel的运行原理

1.AST

首先要理解一个概念：抽象语法树(Abstract Syntax Tree)，Babel本质上就是操作AST来完成代码的转译。那什么是AST呢？可以移步看这篇文章：AST抽象语法树————最基础的javascript重点知识，99%的人根本不了解。我就不献丑了。

2.编译过程

那了解AST是什么以后，就可以理解Babel的工作原理了。上面说过，Babel是一个编译器，所以它的工作过程可以分为三部分：

  1. Parse(解析)：将源代码转换为AST
  2. Transform(转换)：对AST进行遍历修改
  3. Generator(代码生成)：将第二步经过转换过的AST生成新的代码。

解决CSS3动画卡顿

Posted on 2020-02-26

写在前面

最近做的一个小动画，只是简单的文字平移，却出现了卡顿现象，本文内容大部分来源于网上的博客，demo部分则是自己实践的。所以学习前端，实践真的很重要。

为什么会卡顿？

浏览器是单线程运行的。所以我们先要明确以下几下概念：单线程，主线程和合成线程。

一般情况下，主线程：

  1. 运行JavaScript
  2. 计算HTML元素的CSS样式
  3. 页面的布局
  4. 将元素绘制到一个或多个位图中
  5. 将这些位图交给合成线程

相应地，合成线程负责：

  1. 通过GPU将位图绘制到屏幕上
  2. 通知主线程更新页面中可见或即将变成可见的部分的位图
  3. 计算出页面中哪个部分是可见的
  4. 计算出当你在滚动页面时哪部分是即将变成可见的
  5. 当你滚动页面时将相应位置的元素移动到可视区域

那么为什么会造成动画卡顿呢？

原因是主线程和合成线程的调度不合理。

在使用height,width,margin,padding作为transition的值时，会造成浏览器主线程的工作较重。例如，从margin-left: -20px渲染到margin-left:0,主线程需要计算样式margin-left:19px,margin-left:-18px…一直到margin-left:0,而且每一次主线程计算样式后，合成进程都需要绘制到GPU然后渲染到屏幕上，前后总共进行20次主线程渲染，20次合成进程渲染，20次+20次，总共40次计算。

主线程的渲染流程，可以参考浏览器渲染网页的流程：

  1. 使用 HTML 创建文档对象模型（DOM）
  2. 使用 CSS 创建 CSS 对象模型（CSSOM）
  3. 基于 DOM 和 CSSOM 执行脚本（Scripts）
  4. 合并 DOM 和 CSSOM 形成渲染树（Render Tree）
  5. 使用渲染树布局（Layout）所有元素。计算节点位置和尺寸信息
  6. 渲染（Paint）所有元素。在实际的渲染中，浏览器会尽可能在多个层上渲染，这个层类似于PS里的图层概念

也就是说，主线程每次都需要执行Scripts, Render Tree, Layout和Paint这四个阶段。

而使用transform，例如transform:translate(-20px,0)到transform:translate(0,0)，主线程只需要进行一次transform:translate(-20px,0)到transform:translate(0,0)，然后合成线程将-20px转换到0px，这样的话，总计1+20次。

为什么使用transform更加平滑？

可能有人会问，为什么使用transform会更加平滑？

原因在于，在使用transform做动画时，tranform实现的动画是与合成器线程相关的，不需要等待主线程样式计算或者JS执行，计算速度是很快的，也就是避免了Paint；而使用height,width,margin和padding时，导致布局和绘制的调整，主线程需要重新计算样式并执行JS，计算速度就自然慢了。

我们看看合成器线程干了什么：

Composite：将每个渲染层合并，生成最终的一层渲染画面。Paint阶段，每个层各自渲染，并不关心与其他层之间的关系。Composite就需要将这些层以正确的关系合成，有点像PS的导出PNG。合成发生在GPU上

所以包括平时说的讲translate2D变3D，实际上是触发了层提升，使得响应的元素可以在独立层上与其他层并行处理，间接提升了渲染性能。至于所说的GPU加速，也只是因为被新建了一个层。

似乎提升层来提升性能是个很不错的玩法，但是，你的硬件不是无限的，每多一个层，就会多一份内存，因此，控制层数，也是很重要的性能提升。

总结

正因为transform和opacity可以跳过Paint，并且可以在某种形式下告知浏览器优先以GPU来渲染，才有了现代CSS动画推崇优先使用transform，避免使用position、height等属性的变更来处理动画。一些流行的动画库，如iScroll、Swiper.js等，都是使用transform来处理位置偏移，而非top、left等，就是因为性能更高。

一、伪类和伪元素的定义

1.伪类

CSS3给出的定义是：

伪类存在的意义是为了通过选择器，格式化DOM树以外的信息以及不能被常规CSS选择器获取到的信息。

通过上面的概念我们知道伪类的功能有两种：

  1. 格式化DOM树以外的信息，比如<a>标签的:link和:visited等。这些信息不存在于DOM树当中。
  2. 不能被常规CSS选择器获取到的信息。比如：要获取第一个子元素，我们无法用常规的CSS选择器获取，但可以通过:first-child来获取到。

伪元素可以创建一些文档语言无法创建的虚拟元素。比如：文档语言没有一种机制可以描述元素内容的第一个字母或第一行，但伪元素可以做到(::first-letter、::first-line)。同时，伪元素还可以创建源文档不存在的内容，比如使用::before或::after。

XSS攻击

Posted on 2020-01-27

一、什么是XSS？

XSS全称是Cross Site Scripting即跨站脚本，XSS攻击通常指通过利用网页开发时留下的漏洞，通过巧妙的方法注入恶意指令到页面，使用户加载并执行攻击者恶意制造的网页程序。

二、危害

  1. 盗取各类用户账号
  2. 读取、篡改、添加、删除敏感数据
  3. 盗取资料
  4. 非法转账
  5. 强制发送电子邮件

三、XSS类型

  1. 持久型跨站

持久型XSS，也叫存储型XSS，主要是将XSS代码发送到服务器。最典型的案例是留言板XSS。

用户提交一条包含XSS代码的留言到数据库，当目标用户查看留言板时，留言内容会从数据库查询出来并展示，浏览器发现有XSS代码，会当作正常的HTML与JS解析执行，触发XSS攻击。

  1. 非持久型跨站

TCP为什么是三次握手四次挥手

Posted on 2020-01-23

一、写在前面

关于三次握手的问题，我看了很多博客，但是说法不一，谢希仁著《计算机网络》第四版中，讲“三次握手”的目的是：

防止已失效的连接请求报文段又传送到服务器端，造成资源的浪费。

但实际上，这个只能算表因，并不涉及本质。

二、TCP三次握手过程

TCP报文格式

字段含义

字段名 含义 作用
Seq 序列号 用来标识从TCP源端向目的端发送的字节流
ack 确认号 只有Ack标志位为1时，确认序号字段才有效。ack = Seq + 1，若同意连接，则响应报文中应该使SYN=1，ACK=1
SYN 标志位 在连接建立时用来同步序号。当SYN=1，ACK=0，表明是连接请求报文
FIN 标志位 释放一个新连接,当FIN=1,表明此报文的发送方数据已经发送完毕
RST 标志位 重置连接
ACK 标志位 仅当ACK=1时，确认号字段才有效
URG 标志位 表明紧急指针有效，告诉系统此报文段中有紧急数据
PSH 标志位 接收方应该尽快将这个报文交给应用层

注意：

  1. 不要将确认序号ack与标志位中的ACK搞混
  2. 确认方Ack=发起方Seq+1，两端配对

三次握手过程

三、为什么TCP需要握手这个操作

在解答为什么TCP需要三次握手，而不是两次之前，首先需要回答的问题是：为什么需要握手这个操作，能不能不握手？

这里引出了TCP和UDP的一个基本区别：TCP是可靠通信协议，而UDP是不可靠通信协议。

  • TCP的可靠性含义：接收方收到的数是完整、有序、无差错的。
  • UDP不可靠性含义：接收方接收到的数据可能存在部分丢失，顺序也不一定能保证。

TCP可靠传输的精髓

TCP协议为了实现可靠传输，通信双方需要判断自己已经发送的数据包是否都被接收方收到，如果没收到，就需要重新发送。因此，为了实现这个需求。就引出序号Seq和确认号Ack的使用。

发送方在发送数据包(假设大小为10byte)，同时会发送一个序列号(假设为500)，那么接收方收到这个数据包以后，就回复一个确认号(500+10=510),告诉发送方：“我已经收到了你的数据包，你可以发送下一个数据包，序号从510开始。”

这样发送方就可以知道哪些数据被接收到，哪些数据没被接收到，需要重发。

四、为什么TCP是三次握手？

总结上文内容，一句话概括，TCP连接握手，握的是什么？

是通信双方数据原点的序列号！

五、为什么TCP是四次挥手

因为服务端在LISTEN状态下，收到建立连接请求的SYN报文后，把ACK和FIN放在一个报文中发送给客户端。而关闭连接时，当收到对方的FIN报文时，仅仅表示对方不再发送数据了但是还能接收数据。此时，服务端也未必将全部数据都发送给对方了。所以我们可以不必立即close，也可以发送一些数据，再发送FIN报文给对方来表示同意现在关闭连接。所以在关闭连接时，FIN和ACK是分开发送的。

instanceof 实现原理

五、当我们在new一个对象时，我们在做什么

当执行new func的时候，执行过程如下：

  1. 首先创建一个新的空对象，obj
  2. obj.[[prototype]] = func.prototype
  3. 执行构造函数中的代码，构造函数中的this指向该对象
  4. 返回对象
